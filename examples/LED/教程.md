## 1. 项目简介

本教程将带你用 **Rust 语言** 在 **ESP32-S3** 芯片上完成一个最经典的嵌入式入门实验 —— **LED 闪烁**。

你将学到：

- Rust 裸机（`no_std`）开发的基本概念
- `esp-hal` 硬件抽象层的使用方法
- GPIO 输出控制（推挽输出、电平设置、toggle 翻转）
- RTT 调试输出的配置
- 从零编译并烧录到 ESP32-S3 开发板

> 💡 本项目属于 [Rust_ESP32_Dome](https://github.com/cx693/Rust_ESP32_Dome) 仓库的 LED 示例。

## 2. 硬件介绍

### 嘉立创 ESP32-S3R8N8 开发板参数

| 参数 | 值 |
|---|---|
| 芯片 | ESP32-S3 |
| Flash | 8MB（N8 = 8MB NOR Flash） |
| PSRAM | 8MB（R8 = 8MB Octal PSRAM） |
| 板载 LED 引脚 | **GPIO48** |
| LED 类型 | **Active-Low**（低电平点亮） |
| USB 接口 | Type-C（内置 USB-JTAG 调试） |

### Active-Low 是什么意思？

```
┌──────────────────────────────────────┐
│         LED 电路示意图               │
│                                      │
│   VCC ──── R(1kΩ) ──── LED ──── GPIO48
│                                      │
│   GPIO48 = Low  → 电流流通 → LED 亮  │
│   GPIO48 = High → 电流截止 → LED 灭  │
└──────────────────────────────────────┘
```

很多开发板把 LED 的阳极接 VCC、阴极接 GPIO，所以 **拉低 GPIO 才能点亮 LED**。这就是 "Active-Low"。

## 5. 核心依赖解析

打开 `Cargo.toml`，我们只需要关注 LED 闪烁相关的几个依赖：

```toml
# 硬件抽象层 —— 最核心的依赖，提供 GPIO、定时器、外设控制等
esp-hal = { version = "~1.1.0", features = ["esp32s3", "unstable"] }

# ESP-IDF bootloader 支持（提供 esp_app_desc! 宏）
esp-bootloader-esp-idf = { version = "0.5.0", features = ["esp32s3"] }

# RTT 调试输出（类似 printf，通过调试器查看）
rtt-target = "0.6.2"
panic-rtt-target = "0.2.0"
```

> ⚠️ `esp-rtos`、`esp-radio`、`bleps`、`smoltcp` 等是 WiFi/BLE 相关依赖，LED 闪烁用不到，但保留不影响编译。

## 6. 完整源码 + 逐行注释

### 📄 `src/bin/main.rs`

```rust
// ============================================================
// ESP32-S3 LED 闪烁示例
// 功能: 每隔 500ms 切换一次板载 LED（GPIO48）的亮/灭状态
// 硬件: ESP32-S3 开发板，板载 LED 为低电平点亮（active-low）
// ============================================================

#![no_main] // 告诉编译器不使用标准 main 入口，由 esp_hal 提供
#![no_std]  // 裸机环境不使用标准库（std），只用核心库（core）

use esp_bootloader_esp_idf;
use esp_hal::{
    delay::Delay,                           // 硬件定时器延时
    gpio::{Level, Output, OutputConfig},    // GPIO 输出控制
    main,                                   // 程序入口宏
};
use rtt_target::{rprintln, rtt_init_print}; // RTT 调试输出
use panic_rtt_target as _;                  // panic 时通过 RTT 输出错误信息

// 注册 ESP-IDF bootloader 的应用描述符（固件元信息，必须调用）
esp_bootloader_esp_idf::esp_app_desc!();

/// 程序入口
/// `#[main]` 由 esp_hal 提供，替代标准库的 main，完成芯片初始化后调用此函数
/// 返回类型 `-> !` 表示永不返回（嵌入式程序通常是一个无限循环）
#[main]
fn main() -> ! {
    // 初始化 RTT 调试输出（通过 J-Link/ESP-Prog 等调试器查看）
    rtt_init_print!();
    rprintln!("LED 闪烁示例启动");

    // 初始化 HAL（硬件抽象层），获取所有外设的控制权
    let peripherals = esp_hal::init(esp_hal::Config::default());

    // 配置 GPIO48 为推挽输出，初始电平为 Low
    // ESP32-S3 许多开发板的板载 LED 是 active-low（低电平点亮）
    // 所以 Level::Low = LED 亮，Level::High = LED 灭
    let mut led = Output::new(peripherals.GPIO48, Level::Low, OutputConfig::default());

    // 创建延时器（基于硬件定时器，非忙等，CPU 可以低功耗等待）
    let delay = Delay::new();

    loop {
        // toggle() 翻转引脚电平：High → Low，Low → High
        // 比手动 set_level(if ...) 更简洁直观
        led.toggle();

        // 读取当前引脚状态并打印
        // is_set_high() 返回 true 表示引脚输出高电平
        // 由于 active-low：高电平 = LED 灭，低电平 = LED 亮
        if led.is_set_high() {
            rprintln!("LED 灭");
        } else {
            rprintln!("LED 亮");
        }

        // 延时 500 毫秒（非忙等，比 Instant::now() 的忙等循环更高效）
        delay.delay_millis(500);
    }
}
```

## 7. 关键知识点详解

### 7.1 `#![no_main]` 和 `#![no_std]`

| 属性 | 含义 |
|---|---|
| `#![no_std]` | 不使用标准库 `std`，只用核心库 `core`。嵌入式没有操作系统，无法使用 `std` 中的文件、网络、线程等功能 |
| `#![no_main]` | 不使用 Rust 标准的 `main` 函数入口。由 `esp_hal` 的 `#[main]` 宏提供芯片初始化 + 入口 |

### 7.2 GPIO 输出模式

```rust
Output::new(pin, initial_level, config)
```

| 参数 | 说明 |
|---|---|
| `pin` | GPIO 引脚，如 `peripherals.GPIO48` |
| `initial_level` | 初始电平：`Level::High` 或 `Level::Low` |
| `config` | `OutputConfig::default()` 默认为推挽输出（Push-Pull），20mA 驱动电流 |

### 7.3 `toggle()` vs `set_level()`

```rust
// ❌ 写法一：繁琐，需要额外变量
let mut on = true;
led.set_level(if on { Level::Low } else { Level::High });
on = !on;

// ✅ 写法二：简洁，一行搞定
led.toggle();
```

### 7.4 `Delay` vs `Instant` 延时对比

```rust
// ❌ 忙等（Busy-Wait）—— CPU 100% 空转，功耗高
let now = Instant::now();
while now.elapsed().as_millis() < 500 {}

// ✅ 硬件定时器延时 —— CPU 可进入低功耗
let delay = Delay::new();
delay.delay_millis(500);
```

### 7.5 RTT 调试输出

**RTT**（Real-Time Transfer）是 Segger 提出的调试输出协议，通过 J-Link 等调试器实时查看打印信息，比串口更快更稳定。

```
┌──────────┐    USB/JTAG    ┌──────────┐
│  ESP32-S3 │ ──────────────→│   PC     │
│  rprintln!│    RTT 通道    │ probe-rs │
└──────────┘                └──────────┘
```

### 7.6 `-> !` 不返回类型

```rust
fn main() -> ! {
    loop {
        // 永远不会退出
    }
}
```

`-> !` 是 Rust 的 **never type**，表示函数永远不会返回。嵌入式程序的 `main` 通常是一个无限循环，所以用 `-> !` 标记。

## 10. 常见问题 FAQ

### ❓ 编译报错：`no Delay in time`

```
error[E0432]: unresolved import `esp_hal::time::Delay`
```

**原因**：`Delay` 在 `esp_hal::delay` 模块下，不在 `time` 里。

**修复**：
```rust
// ❌ 错误
use esp_hal::time::Delay;

// ✅ 正确
use esp_hal::delay::Delay;
```

---

### ❓ 编译报错：`unresolved import esp_hal::gpio::Output`

**原因**：`esp-hal` 版本过低（低于 v1.0.0），旧版 API 不同。

**修复**：确保 `Cargo.toml` 中：
```toml
esp-hal = { version = "~1.1.0", features = ["esp32s3", "unstable"] }
```

---

### ❓ 烧录报错：`probe not found`

**原因**：调试器未连接或驱动问题。

**排查步骤**：
1. 检查 USB 线是否支持数据传输（非纯充电线）
2. macOS 下执行 `system_profiler SPUSBDataType` 查看是否识别到设备
3. 尝试 `probe-rs list` 查看已连接的调试器

---

### ❓ LED 不亮

**排查清单**：
1. 确认你的开发板 LED 连接的是 **GPIO48**（不同开发板可能不同）
2. 确认 LED 是 **Active-Low**（低电平点亮）
3. 检查固件是否烧录成功（RTT 终端是否有输出）

---

### ❓ rust-analyzer 报错但能正常编译

**原因**：rust-analyzer 默认用 stable 工具链分析，但本项目需要 `esp` 工具链。

已在 `.vscode/settings.json` 中配置：
```json
{
    "rust-analyzer.server.extraEnv": {
        "RUSTUP_TOOLCHAIN": "stable"
    }
}
```

> 💡 如果仍然报错，不影响 `cargo build` 和烧录，可以忽略 IDE 的红色波浪线。
