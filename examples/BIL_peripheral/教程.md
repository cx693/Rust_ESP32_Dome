## 一、项目简介

本教程将带你从零开始，用 **Rust** 在 **ESP32-S3** 上实现一个 **BLE（蓝牙低功耗）电池服务外设**。

完成后，你的 ESP32-S3 将能：

- 📡 **广播蓝牙信号**，手机可搜索到名为 `BLE-ESP32` 的设备
- 🔋 **提供电池服务**（Battery Service, UUID: 0x180F），手机可读取电量
- 📲 **实时推送通知**，每 2 秒向手机推送电量变化（模拟电池放电：100% → 0%）
- 📶 **读取信号强度**（RSSI），实时监测连接质量

### 为什么用 Rust 写嵌入式？

| 对比项 | C（传统方式） | Rust |
|--------|-------------|------|
| 内存安全 | 手动管理，容易出错 | 编译器保证，零成本抽象 |
| 并发安全 | 需要小心处理锁和竞态 | 所有权系统，编译期防竞态 |
| 生态工具 | ESP-IDF 成熟 | esp-rs 快速发展，已支持 ESP32-S3 |
| 学习曲线 | 低（对嵌入式而言） | 较高，但一次学会终身受益 |

## 四、核心概念速查

在看代码之前，先理解几个关键概念：

### 4.1 BLE 基础

```
┌─────────────────────────────────────────────────────┐
│  BLE 通信模型                                        │
│                                                      │
│  ┌──────────┐         广播 / 连接         ┌────────┐│
│  │ 外设      │  ◄──────────────────────►  │ 中央    ││
│  │(Peripheral)│                            │(Central)││
│  └──────────┘                             └────────┘│
│       │                                      手机/PC │
│       ▼                                              │
│  ┌──────────────────────────────────────┐           │
│  │  GATT 服务器                          │           │
│  │  ┌────────────────────────────────┐  │           │
│  │  │  服务 (Service)                 │  │           │
│  │  │  UUID: 0x180F (电池服务)        │  │           │
│  │  │  ┌──────────────────────────┐  │  │           │
│  │  │  │  特征 (Characteristic)    │  │  │           │
│  │  │  │  - level:  电量 (u8)      │  │  │           │
│  │  │  │  - status: 状态 (bool)    │  │  │           │
│  │  │  └──────────────────────────┘  │  │           │
│  │  └────────────────────────────────┘  │           │
│  └──────────────────────────────────────┘           │
└─────────────────────────────────────────────────────┘
```

- **GATT**：通用属性协议，定义数据如何组织和传输
- **Service（服务）**：功能的逻辑分组（如"电池服务"）
- **Characteristic（特征）**：服务中的具体数据项（如"电量值"）
- **Notify（通知）**：外设主动推送数据给中央设备（无需轮询）
- **RSSI**：信号强度指示，单位 dBm，越接近 0 信号越强

## 五、代码逐行解析

### 5.1 文件头与导入

```rust
//! BLE 电池服务外设示例（Battery Service Peripheral）
//!
//! 功能概述：
//! 1. 初始化 ESP32-S3 硬件和 BLE 控制器
//! 2. 以随机地址广播，等待中央设备（Central）连接
//! 3. 连接后提供 GATT 电池服务（电量读取/通知 + 自定义状态特征）
//! 4. 每 2 秒向已连接的中央设备推送电量值，并读取 RSSI 信号强度

#![no_std]  // 嵌入式环境：不使用标准库（没有堆、文件系统、网络等）
#![no_main] // 入口函数由 esp_rtos 管理，不需要标准 main

use embassy_executor::Spawner;
use embassy_futures::{join::join, select::select}; // 异步组合子
use embassy_time::Timer;                            // 异步定时器
use esp_alloc as _;          // 堆分配器（BLE 协议栈需要动态内存）
use esp_backtrace as _;      // panic 时的回溯处理
use esp_hal::{
    clock::CpuClock,
    interrupt::software::SoftwareInterruptControl,
    timer::timg::TimerGroup,
};
use esp_radio::ble::controller::BleConnector; // BLE 硬件连接器
use rtt_target::{rprintln, rtt_init_print};   // RTT 调试输出
use trouble_host::prelude::*;                 // BLE 协议栈核心类型
```

> **`#![no_std]` 是什么？**
> 标准库（std）依赖操作系统，嵌入式设备没有 OS，所以用 `no_std`。
> 这意味着不能用 `Vec`、`String`、`println!` 等，但可以用 `core` 和 `alloc` 提供的类型。

### 5.2 常量定义

```rust
/// 最大同时连接数（外设通常只需 1 个）
const CONNECTIONS_MAX: usize = 1;
/// L2CAP 通道数：1 个信令通道 + 1 个 ATT 数据通道
const L2CAP_CHANNELS_MAX: usize = 2;
```

> **L2CAP 是什么？**
> BLE 的传输层协议，类似 TCP/IP 中的 TCP。
> 每个连接至少需要 2 个通道：信令通道（控制）+ ATT 数据通道（业务数据）。

### 5.3 GATT 服务定义

```rust
/// GATT 服务器：包含所有服务的集合
#[gatt_server]
struct Server {
    battery_service: BatteryService,
}

/// 电池服务（UUID: 0x180F —— BLE 标准电池服务）
#[gatt_service(uuid = service::BATTERY)]
struct BatteryService {
    /// 电量特征：可读 + 可通知，范围 0~100，初始值 10
    #[descriptor(uuid = descriptors::VALID_RANGE, read, value = [0, 100])]
    #[descriptor(uuid = descriptors::MEASUREMENT_DESCRIPTION, read, value = "Battery Level")]
    #[characteristic(uuid = characteristic::BATTERY_LEVEL, read, notify, value = 10)]
    level: u8,

    /// 自定义状态特征：可读 + 可写 + 可通知
    #[characteristic(uuid = "408813df-5dd4-1f87-ec11-cab001100000", write, read, notify)]
    status: bool,
}
```

> **`#[gatt_server]` 宏做了什么？**
> 它是 `trouble-host` 提供的过程宏，会自动生成：
> - GATT 属性表（Attribute Table）
> - 读写请求的路由逻辑
> - `Server::new_with_config()` 构造函数
> - `server.get(&characteristic)` / `server.set(&characteristic, value)` 方法
>
> 你只需要声明结构体，宏帮你搞定底层细节。

### 5.4 入口函数

```rust
#[esp_rtos::main] // Embassy 异步入口宏
async fn main(_s: Spawner) {
    // 1) 初始化 RTT 调试输出（通过 probe-rs 在电脑端查看）
    rtt_init_print!();
    rprintln!("=== BLE 电池外设启动 ===");

    // 2) 初始化硬件：外设、CPU 时钟拉满
    let peripherals = esp_hal::init(
        esp_hal::Config::default().with_cpu_clock(CpuClock::max())
    );

    // 3) 分配堆内存（BLE 协议栈需要动态内存，72KB）
    esp_alloc::heap_allocator!(size: 72 * 1024);

    // 4) 启动 esp-rtos 异步运行时
    let timg0 = TimerGroup::new(peripherals.TIMG0);
    let sw_int = SoftwareInterruptControl::new(peripherals.SW_INTERRUPT);
    esp_rtos::start(timg0.timer0, sw_int.software_interrupt0);

    // 5) 创建 BLE 控制器
    let connector = BleConnector::new(peripherals.BT, Default::default()).unwrap();
    let controller: ExternalController<_, 1> = ExternalController::new(connector);

    // 6) 进入 BLE 主循环
    ble_bas_peripheral_run(controller).await;
}
```

> **初始化流程图：**
>
> ```
> rtt_init_print!()          ← 调试输出就绪
>        │
> esp_hal::init()            ← 外设初始化
>        │
> esp_alloc::heap_allocator! ← 堆内存分配
>        │
> esp_rtos::start()          ← 异步运行时启动
>        │
> BleConnector::new()        ← BLE 硬件连接
>        │
> ble_bas_peripheral_run()   ← 进入主循环
> ```

### 5.5 BLE 主循环

```rust
async fn ble_bas_peripheral_run<C: Controller>(controller: C) {
    // 使用固定随机地址（实际产品中应从硬件 MAC 地址生成）
    let address = Address::random([0xff, 0x8f, 0x1a, 0x05, 0xe4, 0xff]);

    // 创建协议栈资源
    let mut resources: HostResources<DefaultPacketPool, CONNECTIONS_MAX, L2CAP_CHANNELS_MAX> =
        HostResources::new();
    let stack = trouble_host::new(controller, &mut resources)
        .set_random_address(address);

    // 拆分出 peripheral（广播管理器）和 runner（底层事件驱动）
    let Host { mut peripheral, runner, .. } = stack.build();

    // 创建 GATT 服务器
    let server = Server::new_with_config(GapConfig::Peripheral(PeripheralConfig {
        name: "BLE-ESP32",
        appearance: &appearance::power_device::GENERIC_POWER_DEVICE,
    }))
    .unwrap();

    // join: 同时运行两个任务
    //   - ble_task:  底层 BLE 事件循环（必须一直运行）
    //   - 广播循环:  广播 → 连接 → 处理事件 → 断开 → 重新广播
    let _ = join(ble_task(runner), async {
        loop {
            match advertise(&mut peripheral, &server).await {
                Ok(conn) => {
                    rprintln!("中央设备已连接");
                    select(
                        gatt_events_task(&server, &conn),
                        custom_task(&server, &conn, &stack),
                    ).await;
                    rprintln!("连接已断开，重新广播...");
                }
                Err(e) => panic!("[广播] 错误: {:?}", e),
            }
        }
    }).await;
}
```

> **`join` vs `select` 的区别：**
>
> ```
> join(a, b)     ← 同时运行 a 和 b，等两个都结束
> select(a, b)   ← 同时运行 a 和 b，任一结束就全部结束
> ```
>
> 本项目中：
> - `join`：BLE 底层循环 + 广播循环，两个都要一直运行
> - `select`：GATT 事件处理 + 电量通知，任一结束（通常因为断开）就回到广播状态

### 5.6 广播函数

```rust
async fn advertise<'a, 's, C: Controller>(
    peripheral: &mut Peripheral<'a, C, DefaultPacketPool>,
    server: &'s Server<'a>,
) -> Result<GattConnection<'a, 's, DefaultPacketPool>, BleHostError<C::Error>> {
    // 构造广播数据（BLE 广播包最大 31 字节）
    let mut advertiser_data = [0; 31];
    let len = AdStructure::encode_slice(
        &[
            AdStructure::Flags(LE_GENERAL_DISCOVERABLE | BR_EDR_NOT_SUPPORTED),
            AdStructure::ServiceUuids16(&[[0x0f, 0x18]]),  // 电池服务 UUID
            AdStructure::CompleteLocalName(b"BLE-ESP32"),    // 设备名称
        ],
        &mut advertiser_data[..],
    )?;

    // 开始广播
    let advertiser = peripheral.advertise(
        &Default::default(),
        Advertisement::ConnectableScannableUndirected {
            adv_data: &advertiser_data[..len],
            scan_data: &[],
        },
    ).await?;

    rprintln!("[广播] 正在广播，等待中央设备连接...");

    // 等待中央设备连接
    let conn = advertiser.accept().await?.with_attribute_server(server)?;
    rprintln!("[广播] 连接已建立");
    Ok(conn)
}
```

> **BLE 广播包结构（31 字节限制）：**
>
> ```
> ┌──────┬──────────┬───────────────────┐
> │ 长度 │   类型    │      数据          │
> ├──────┼──────────┼───────────────────┤
> │  2   │  Flags   │ 0x06 (可发现+BLE)  │
> │  3   │ UUID列表  │ 0x180F (电池服务)  │
> │ 10   │ 设备名称  │ "BLE-ESP32"       │
> └──────┴──────────┴───────────────────┘
> ```

### 5.7 GATT 事件处理

```rust
async fn gatt_events_task<P: PacketPool>(
    server: &Server<'_>,
    conn: &GattConnection<'_, '_, P>,
) -> Result<(), Error> {
    let level = server.battery_service.level;
    let status = server.battery_service.status;

    let reason = loop {
        match conn.next().await {
            GattConnectionEvent::Disconnected { reason } => break reason,
            GattConnectionEvent::Gatt { event } => {
                match &event {
                    GattEvent::Read(event) => {
                        // 处理读取请求（手机读取特征值时触发）
                        if event.handle() == level.handle {
                            rprintln!("[GATT] 读取电量: {:?}", server.get(&level));
                        } else if event.handle() == status.handle {
                            rprintln!("[GATT] 读取状态: {:?}", server.get(&status));
                        }
                    }
                    GattEvent::Write(event) => {
                        // 处理写入请求（手机写入特征值时触发）
                        if event.handle() == level.handle {
                            rprintln!("[GATT] 写入电量: {:?}", event.data());
                        } else if event.handle() == status.handle {
                            rprintln!("[GATT] 写入状态: {:?}", event.data());
                        }
                    }
                    _ => {}
                };
                // 必须显式接受事件并回复（否则手机会超时）
                match event.accept() {
                    Ok(reply) => reply.send().await,
                    Err(e) => rprintln!("[GATT] 响应失败: {:?}", e),
                };
            }
            _ => {}
        }
    };
    rprintln!("[GATT] 断开原因: {:?}", reason);
    Ok(())
}
```

### 5.8 电量通知任务

```rust
async fn custom_task<C: Controller, P: PacketPool>(
    server: &Server<'_>,
    conn: &GattConnection<'_, '_, P>,
    stack: &Stack<'_, C, P>,
) {
    let mut battery_level: u8 = 100; // 初始电量 100%
    let level = server.battery_service.level;

    loop {
        // 模拟电池放电：每 2 秒减少 1%
        if battery_level == 0 {
            battery_level = 100; // 到 0% 后重新开始（方便循环演示）
        } else {
            battery_level -= 1;
        }

        // 通过 notify 推送电量（手机无需轮询，自动收到）
        rprintln!("[通知] 电量 = {}%", battery_level);
        if level.notify(conn, &battery_level).await.is_err() {
            rprintln!("[通知] 推送失败");
            break;
        }

        // 读取 RSSI 信号强度
        match conn.raw().rssi(stack).await {
            Ok(rssi) => rprintln!("[RSSI] {} dBm", rssi),
            Err(_) => { rprintln!("[RSSI] 读取失败"); break; }
        }

        Timer::after_secs(2).await; // 等待 2 秒
    }
}
```

> **Notify vs Read 的区别：**
>
> | 方式 | 谁发起 | 适用场景 |
> |------|--------|---------|
> | Read | 中央设备（手机） | 偶尔查询 |
> | Notify | 外设（ESP32） | 实时推送（如心率、电量变化） |

## 八、常见问题

### Q1: 编译报错 `cannot find module embassy_sync`

`#[gatt_server]` 过程宏内部会引用 `embassy_sync`、`static_cell`、`heapless`，即使你没有显式 `use` 它们。确保 `Cargo.toml` 中有这三个依赖。

### Q2: `cargo run` 后没有输出

确认使用了 `rtt_init_print!()` 宏，并且使用 `probe-rs` 烧录（不是 `espflash`）。RTT 输出需要调试器支持。

### Q3: 手机搜不到 BLE-ESP32

- 确认 ESP32-S3 的蓝牙天线已连接（嘉立创开发板通常板载天线）
- 检查日志是否显示 `[广播] 正在广播...`
- 有些手机需要开启"蓝牙扫描"权限

### Q4: 连接后很快就断开

- 可能是 RSSI 太差（距离太远）
- 检查日志中的 `[GATT] 断开原因`
- 常见原因：`Remote User Terminated Connection`（用户主动断开）是正常的

## 九、扩展方向

掌握本示例后，可以尝试：

1. **添加更多服务**：心率服务（0x180D）、温度服务（0x1809）
2. **读取真实传感器**：用 ADC 读取电池电压，转换为真实电量百分比
3. **低功耗模式**：连接空闲时进入 Deep Sleep，通过蓝牙唤醒
4. **多连接**：修改 `CONNECTIONS_MAX` 支持多个中央设备同时连接
5. **BLE Mesh**：多个 ESP32 组网，实现智能家居控制
